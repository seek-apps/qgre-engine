## DrGRPO / GRPO

**Current usage:** SPO mode (n=1, persistent EMA baseline). Dr.GRPO available via `loss_type: "dr_grpo"` (removes length normalization, loss_functions.py:158-160). GDPO per-step normalization in advantages.py:153-164. seq-mean-token-sum-norm aggregation.

### 🔴 CRITICAL

- **MAD GRPO: Dr.GRPO introduces verbosity bias.** Dr.GRPO removes per-sequence length normalization, which fixes length bias but implicitly weights updates by completion length, encouraging verbosity. MAD GRPO proposes robust reward scaling via Median Absolute Deviation instead of mean/std. Our GDPO per-step normalization partially mitigates this (normalizes per step, not globally), but worth monitoring output lengths. — [huggingface.co/blog/telcom/mad-grpo](https://huggingface.co/blog/telcom/mad-grpo)
- **GDPO NaN on None/NaN rewards.** ms-swift #8123: GDPO path normalizes each reward function separately using `.mean()` and `.std()`, which propagate NaN if any reward returns None. Our engine: `build_batch_reward_tensors()` zero-fills missing keys, but if a reward_fn returns NaN (not None), it will propagate through GDPO normalization silently. Add NaN guard before GDPO normalization. — [github.com/modelscope/ms-swift/issues/8123](https://github.com/modelscope/ms-swift/issues/8123)
- **verl DrGRPO loss scaling bug.** verl #3741: `seq-mean-token-sum-norm` divides by `loss_mask.shape[-1]` (max possible length) instead of actual sequence count, causing loss to scale with micro_batch_size. Our implementation divides by `n_valid_seqs` and `max(mask.shape[-1], 1)` (loss_functions.py:163), which matches the corrected behavior. No action needed — but verify the divisor stays constant across batches for true Dr.GRPO. — [github.com/verl-project/verl/issues/3741](https://github.com/verl-project/verl/issues/3741)

### 🟡 IMPORTANT

- **GSPO: Sequence-level importance sampling.** Qwen team's GSPO replaces token-level IS ratios with sequence-level, fundamentally reducing variance in MoE and long-sequence settings. Token-level IS weights accumulate multiplicatively with length, causing variance explosion. Our `force_on_policy_ratio=True` avoids this entirely (ratio always 1.0), but if we ever move off on-policy, GSPO's insight applies. — [huggingface.co/blog/NormalUhr/grpo-to-dapo-and-gspo](https://huggingface.co/blog/NormalUhr/grpo-to-dapo-and-gspo)
- **F-GRPO: Focal-loss inspired difficulty-aware scaling.** Down-weights updates on high-success prompts, preventing mode collapse on easy problems. Improves pass@256 by 6+ points. Our SPO prioritized sampling achieves a similar effect (high |advantage| = sample more), but F-GRPO's approach is complementary and lightweight. — [arxiv.org/pdf/2602.06717](https://arxiv.org/pdf/2602.06717)
- **CoRPO: Correctness-biased baseline clipping.** GRPO's group mean can assign positive advantages to incorrect solutions. CoRPO clips the minimum baseline to a correctness threshold. Our SPO mode avoids this (per-prompt EMA, not group mean), but relevant if using GRPO mode. — [arxiv.org/pdf/2511.04439](https://arxiv.org/pdf/2511.04439)
- **TRL loss normalization fix (PR #2881).** TRL's GRPO had per-sequence normalization instead of global. Fixed to global normalization. Our implementation uses global normalization via `global_valid_toks` (loss_functions.py:112). Correct. — [github.com/huggingface/trl/pull/2881](https://github.com/huggingface/trl/pull/2881)
- **Grad norm instability after 10k steps (TRL #2980).** GRPO grad_norm spikes higher over time even as reward improves. Caused by advantage distribution shifting. Monitor grad_norm and clip_ratio. Our `max_grad_norm` clipping (trainer.py:437) handles this. — [github.com/huggingface/trl/issues/2980](https://github.com/huggingface/trl/issues/2980)

### 🟢 NICE-TO-HAVE

- **Noise-corrected GRPO.** Models reward corruption as Bernoulli noise and debiases the learning signal. Up to 6.7pp accuracy gains. Relevant if using imperfect reward models (ours are programmatic, less relevant). — [arxiv.org/pdf/2510.18924](https://arxiv.org/pdf/2510.18924)

### Chinese community insights

- CSDN detailed comparison of DAPO vs DrGRPO confirms they address the same length bias from different angles. DAPO does token-level averaging; DrGRPO removes token averaging entirely. Our GDPO per-step normalization is a third approach that avoids both failure modes. — [yam.gift/2025/08/14/NLP/LLM-Training/2025-08-14-Token-Level-GSPO-GMPO](https://yam.gift/2025/08/14/NLP/LLM-Training/2025-08-14-Token-Level-GSPO-GMPO/)

## KL Estimators (k1/k2/k3)

**Current usage:** k3 hardcoded (trainer.py:104). kl.py implements k1/k2/k3 per Schulman 2020. All three estimators fully implemented.

### 🔴 CRITICAL

- **k3 has biased gradients under policy drift.** "A Comedy of Estimators" (Shah, Bengio et al., arXiv:2512.21852, Dec 2025) — definitive systematic study. When `pi_theta << pi_ref`, k3 assigns unbounded gradient weights. On-policy this is negligible, but once we store real reference logprobs (Plan Item 1), the bias activates as the policy diverges over training. — [arxiv.org/abs/2512.21852](https://arxiv.org/abs/2512.21852)
- **k1-in-reward is the principled loss for reverse KL.** "Rethinking KL Regularization" (Liu et al., ICLR 2026) proves this formally. Under on-policy, k2-as-loss is gradient-equivalent to k1-in-reward. — [openreview.net/forum?id=keCnsHtION](https://openreview.net/forum?id=keCnsHtION)
- **RPG view: k3 = unnormalized KL; off-policy IS mismatch in GRPO.** "On the Design of KL-Regularized PG Algorithms" (Zhang et al., ICLR 2026) — unified framework identifying that GRPO's k3 usage has an importance-weighting mismatch. Specifies conditions under which stop-gradient losses are gradient-equivalent to differentiable surrogates. — [arxiv.org/pdf/2505.17508](https://arxiv.org/pdf/2505.17508)

### 🟡 IMPORTANT

- After Plan Item 1 lands: default to k1-in-reward (detached) or k2-as-loss for on-policy. Keep k3 as option for backward compat. Plan Item 2 (configurable type) handles this.

### Chinese community insights

- CSDN article (April 2025) provides clear comparison with simulation: k3 lowest variance but potential bias; k1 unbiased but high variance; k2 always positive but biased. Matches the papers exactly. — [blog.csdn.net/animate1/article/details/146283006](https://blog.csdn.net/animate1/article/details/146283006)

## LLDS

**Current usage:** Implemented in llds.py, integrated at trainer.py:398-410 (coef=0.05). Three-level gate: trajectory + token + action. Currently dead — old_logprobs == curr_logprobs.

Our implementation matches the paper but all gates are inactive because `old_log_prob == log_prob`. Fix is Plan Item 1. Accepted at ICLR 2026 Workshop (LLA). — [arxiv.org/abs/2512.04220](https://arxiv.org/abs/2512.04220)

## GTPO (Entropy-Weighted Token Credit)

**Current usage:** Not implemented. VPRMs provide between-region credit. GTPO would add within-region token importance.

### 🟡 IMPORTANT

- **GTPO (ByteDance, arXiv:2508.04349, ICLR 2026 under review).** Dynamic Entropy Weighting: `H_i,t / sum(H_k,t)` per token. High-entropy tokens get more gradient. Significantly outperforms DAPO. Composes orthogonally with VPRMs. — [arxiv.org/html/2508.04349v5](https://arxiv.org/html/2508.04349v5)
- **Conflict correction (GTPO trajectory variant).** Prevents penalizing essential formatting tokens shared across positive/negative completions. Uses entropy threshold δ_i gating (ln(2)). — [github.com/winstonsmith1897/GTPO](https://github.com/winstonsmith1897/GTPO)
- **GRPO-S (sequence-level variant)** is lighter and more stable for long CoT. Consider as alternative if per-token entropy is too expensive.

### Chinese community insights

- CSDN detailed walkthrough (Oct 2025) of GTPO/GRPO-S framework confirms entropy-weighting as key driver. Includes comparison with DAPO showing GTPO's superiority. — [blog.csdn.net/2401_85343303/article/details/152555651](https://blog.csdn.net/2401_85343303/article/details/152555651)

## Unsloth + vLLM

**Current usage:** Unsloth ≥2026.3.5, vLLM colocated single GPU. `recreate_engine()` every 50 steps. `gpu_memory_utilization=0.35`. `FastLanguageModel.for_training()` before each forward.

### 🔴 CRITICAL

- **VRAM leak confirmed.** Multiple reports: #3864, #3512, #2470. VRAM grows each step until OOM. No upstream fix. Our `recreate_engine()` every 50 steps is the community-accepted workaround. — [github.com/unslothai/unsloth/issues/3864](https://github.com/unslothai/unsloth/issues/3864)
- **Off-by-one shape mismatch (March 2026, #4181).** TRL's GRPOTrainer `masked_batch_mean` has 25 vs 24 tensor mismatch. This is TRL's bug, not ours — we have custom loss. No action needed, but validates building custom loss. — [github.com/unslothai/unsloth/issues/4181](https://github.com/unslothai/unsloth/issues/4181)
- **Re-tokenization bug in multi-turn (TRL #5224, March 2026).** BPE is not a bijection: decode→re-tokenize produces different IDs. Log-probs no longer correspond to sampled tokens. Our engine avoids this (we work with raw token IDs, never re-tokenize). — [github.com/huggingface/trl/issues/5224](https://github.com/huggingface/trl/issues/5224)

### 🟡 IMPORTANT

- **TRL co-located vLLM pattern (June 2025).** Describes our exact architecture: training and inference share one GPU, alternate modes. Our `set_training_mode()`/`set_inference_mode()` matches. — [huggingface.co/blog/vllm-colocate](https://huggingface.co/blog/vllm-colocate)
- **Parameter sync issue (TRL #3189).** vLLM keeps initial weights throughout training. Our engine saves/loads LoRA after every step (trainer.py:656-659) with hash verification, avoiding this. — [github.com/huggingface/trl/issues/3189](https://github.com/huggingface/trl/issues/3189)
- **Inplace op autograd error (#2610, #2434).** Fixed by calling `for_training()` before each forward. We already do this (trainer.py:340-349). — [github.com/unslothai/unsloth/issues/2610](https://github.com/unslothai/unsloth/issues/2610)

### Chinese community insights

- CSDN guide (March 2025) confirms `gpu_memory_utilization` tuning is essential. Recommends 0.5-0.8 range for single-GPU. Our 0.35 is conservative for 16GB constraint — correct. — [blog.csdn.net/yuanlulu/article/details/146297729](https://blog.csdn.net/yuanlulu/article/details/146297729)

## Reward Shaping

**Current usage:** Programmatic per-quality reward functions (not learned). Partial credit always (never binary). Decomposed by quality dimension. Phase-gated activation.

### 🟡 IMPORTANT

- **RLRR: Relative ranking instead of absolute scores.** Shifts reward shaping from absolute scoring to relative ranking within groups. Mitigates signal sparsity when rewards are identical. Our SPO mode already compares to per-prompt history (not group), which naturally avoids identical-reward sparsity. — [arxiv.org/html/2601.23058v1](https://arxiv.org/html/2601.23058v1)
- **Hard examples matter most.** "Hard Examples Are All You Need" (arXiv:2508.14094): training on hardest examples yields up to 47% more gain. Our prioritized sampling (SPO) already upweights high-|advantage| prompts, achieving a similar effect. — [arxiv.org/html/2508.14094v1](https://arxiv.org/html/2508.14094v1)
- **"Rewarding the Unlikely" (EMNLP 2025).** GRPO has rank bias: highly probable trajectories reinforced, rare ones neglected. Unlikeliness reward up-weights rare correct solutions. Our VPRMs avoid this at the step level (each step scored independently), but worth monitoring at the sequence level. — [aclanthology.org/2025.emnlp-main.1298](https://aclanthology.org/2025.emnlp-main.1298/)

### 🟢 NICE-TO-HAVE

- **Best Practices for Multi-Turn RL (Fireworks AI, Dec 2025).** Practical recipes for reward design, partial vs trajectory-level rewards. Confirms our decomposed per-quality approach. — [fireworks.ai/blog/best-practices-for-multi-turn-RL](https://fireworks.ai/blog/best-practices-for-multi-turn-RL)

### Chinese community insights

- CSDN articles on reward shaping focus on general RLHF reward modeling, not programmatic rewards. No specific findings applicable to our domain-specific reward functions.

## Summary — CRITICAL Items

1. **Store generation logprobs (Plan Item 1).** Without it: KL=0, LLDS dead, no drift detection. Blocking.
2. **k3 bias activates once logprobs stored (Plan Item 2).** Make KL type configurable before/alongside Item 1. Default k1-in-reward or k2-as-loss.
3. **VRAM leak workaround (recreate_engine) is correct.** No upstream fix. Keep it.
4. **GDPO NaN guard needed.** If any reward returns NaN, GDPO normalization silently fails. Add `torch.nan_to_num()` before normalization.
5. **MLflow 3.x breaks model logging API.** We don't use `log_model()` yet, but pin `mlflow<3.0` or test before upgrading.
6. **Scaf-GRPO validates scaffold fading.** Read the paper before implementing Plan Item 7.
7. **Archer validates region-specific KL design.** Our THINK/FORMAT/STEP multipliers match their dual-token approach.
